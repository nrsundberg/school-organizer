## What 0d.2 covers

The signup → trial → checkout flow is the only critical path that touches
money. A regression here ("Start trial does nothing", "Manage billing 500s")
silently loses customers, and nothing in the smoke sweep nor the unit suite
catches it today.

Implementation details have shifted since the parent 0d spec was written:

- The signup form does **not** redirect to Stripe Checkout for the default
  path. `app/routes/auth/signup.tsx` runs `ensureOrgForUser`, lands the org
  as `TRIALING` for 30 days, and redirects to the tenant board.
- `shouldStartCheckoutAfterSignup` only fires when the user **explicitly**
  picks a self-serve plan from `/pricing` (planSelectionSource === "explicit").
  When that fires, the action does call `createCheckoutSessionForOrg` and
  redirects to `https://checkout.stripe.com/...` directly.
- The independent Stripe entry point lives at `POST /api/billing/checkout`
  (admin Billing page + `/pricing` Form action).

So "signup → trial → checkout" is two redirects in two routes. The new spec
exercises both legs without driving Stripe's hosted UI.

### What landed in `e2e/flows/signup-to-paid.spec.ts` (~423 LOC, header
comment included)

Three test cases, organized into two `describe` blocks:

1. **Trial leg** — `default-plan signup creates a TRIALING org and lands on
   the tenant board`. Drives the real 3-step signup UI on the marketing host
   with `?plan=district`. The `DISTRICT` plan goes through
   `isSelfServeBillingPlan() === false`, so even with
   `planSelectionSource === "explicit"` the flow takes the no-card branch
   in `app/routes/auth/signup.tsx` (no Stripe call, redirect to tenant
   board). Asserts:
     - URL after step-3 submit is on `<slug>.localhost:8787` (not /signup).
     - `Org.status === "TRIALING"`, `billingPlan === "DISTRICT"`,
       `trialEndsAt` is 28-32 days out.
     - `User`/`Account`/`Session` rows created, captured for cleanup.
   Cleanup: per-spec `afterEach` hook runs the same FK-ordered DELETE list
   used by `seeded-tenant.ts § teardownSeedRows`, scoped to the org/user
   ids the test created. Best-effort + try/catch per the spec quality rule.

2. **Checkout leg (happy path)** — `POST /api/billing/checkout redirects to
   checkout.stripe.com for an authed admin`. Uses the seeded-tenant fixture
   (admin already logged in) + `request.post` with `maxRedirects: 0`.
   Asserts:
     - 302/303 status with `Location: https://checkout.stripe.com/...`.
     - `Org.stripeCustomerId` lazily populated (starts with `cus_`) as a
       side-effect of `createCheckoutSessionForOrg`.
   Conditionally `test.fixme()`'d when `.dev.vars` lacks
   `STRIPE_SECRET_KEY`. The check reads `.dev.vars` from the same cwd
   `wrangler dev` reads it from, so the gate condition tracks reality
   without an extra env var to keep in sync. Per the research spec § "Why
   no E2E_BYPASS_STRIPE flag", the bypass flag was deliberately NOT added
   — a synthetic redirect doesn't catch the regressions we actually fear
   (renamed price IDs, wrong destination), and both surface from the real
   call.

3. **Checkout leg (anonymous → /login)** — `POST /api/billing/checkout
   requires an authed admin`. No cookie header. Asserts the response
   redirects to a `/login`-shaped path (and crucially **not** to
   `checkout.stripe.com`) — a regression gate against the action ever
   exposing checkout to anonymous visitors. Runs whether or not Stripe is
   configured.
