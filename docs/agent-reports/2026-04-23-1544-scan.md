## Findings

### [P0] `/api/healthz` redirects to `/login` instead of returning JSON

- **Where:** `GET /api/healthz`
- **What:** The health endpoint returns `302 Location: /login?next=/api/healthz` to unauthenticated callers, breaking the documented contract (`{ "ok": true }` JSON) that the smoke suite and external uptime monitors both rely on.
- **Evidence:**
  - `curl -sSI https://school-organizer-staging.sundbergne.workers.dev/api/healthz` → `HTTP/2 302` with `location: /login?next=%2Fapi%2Fhealthz`.
  - Playwright `request.get(…/api/healthz, { maxRedirects: 0 })` → status `302`, body empty (0 bytes).
  - The existing smoke suite has two cases that assert the opposite (`e2e/smoke.spec.ts:105` — "GET /api/healthz returns 200 JSON with ok: true" — and `e2e/smoke.spec.ts:123` — "does not 5xx"). Both will fail against the current deploy once the runner can execute.
- **Reproduction (Playwright):**
  ```ts
  const resp = await request.get(
    'https://school-organizer-staging.sundbergne.workers.dev/api/healthz',
    { maxRedirects: 0 },
  );
  expect(resp.status()).toBe(200);
  expect(await resp.json()).toEqual({ ok: true });
  ```
- **Suspected cause:** The same middleware gap that hits `/status` (below) — `publicMarketingPath` in `app/domain/utils/global-context.server.ts` does not list `/api/*` (or at least `/api/healthz`), so unauthenticated callers on the marketing host get kicked to login by the gating layer before the API handler runs.
- **Status:** open

### [P1] `/status` still redirects to `/login?next=/status` (known bug, not fixed)

- **Where:** `GET /status`
- **What:** The public status page 302s to login for unauthenticated visitors; it should render as a marketing route.
- **Evidence:**
  - `curl -sSI …/status` → `HTTP/2 302 location: /login?next=%2Fstatus`.
  - Smoke run captured: `{ status: 200, finalUrl: '…/login?next=%2Fstatus', notes: ['expected self but landed on /login'] }` — the landmark (any heading) rendered because the login page has one, so the underlying bug is hidden from a naive 200 check.
- **Reproduction (Playwright):**
  ```ts
  const resp = await page.goto('/status');
  expect(new URL(page.url()).pathname).toBe('/status'); // fails — lands on /login
  ```
- **Suspected cause:** Documented in `e2e/smoke-routes.ts:75-84` with a `fixme`: "BUG: /status is not in middleware publicMarketingPath; it 302s to /login". Still reproducing on master SHA `76c3dd0`. The P0 `/api/healthz` finding is almost certainly the same root cause — fixing both likely means one edit to `publicMarketingPath`.
- **Status:** open

### [P3] `net::ERR_ABORTED` on `youtube-nocookie.com` embed during `/` → `/pricing` navigation

- **Where:** Signup critical-path flow, first hop from `/` to `/pricing`.
- **What:** Three aborted requests to `https://www.youtube-nocookie.com/embed/M7lc1UVf-VE` recorded by `page.on('requestfailed', …)` during the flow.
- **Evidence:** `failure: "net::ERR_ABORTED"`, no 4xx/5xx observed on any page request. The homepage embeds a YouTube demo video; when the user navigates away before the iframe finishes loading, the browser cancels the in-flight request — this is the `ERR_ABORTED` code path, not a functional failure.
- **Suspected cause:** Benign navigation-cancel. Listed here only so the fixer doesn't file it separately if they re-run.
- **Status:** suspected / no action
