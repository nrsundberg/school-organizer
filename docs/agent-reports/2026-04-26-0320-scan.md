### [P1] React error #418 (hydration mismatch) on every public route

- **Where:** every route in the desktop sweep — confirmed on `/`,
  `/pricing`, `/faqs`, `/blog`, `/blog/parent-communication-…`,
  `/guides`, `/guides/setting-up-your-first-drill-template`,
  `/status`, `/login`, `/signup?plan=car-line`, `/forgot-password`,
  `/reset-password`, `/admin` (post-redirect to `/login?next=…`),
  `/admin/users`, `/admin/children`, `/admin/billing`,
  `/admin/branding`, `/admin/fire-drill`, `/create/homeroom`,
  `/create/student`, `/homerooms`, `/viewer-access` (post-redirect to
  `/`). 22/22 probed routes raise exactly one occurrence per page.
- **What:** Every route raises a single uncaught
  `Minified React error #418` (text-content hydration mismatch:
  https://react.dev/errors/418) in the browser. The page still
  renders (React falls back to client-rendering the mismatched
  subtree), but the smoke contract `pageErrors.length === 0` is
  violated everywhere — `e2e/smoke-routes.ts:smokeOne` would fail on
  every route once the test runner is unblocked. Hydration mismatch
  also defeats SSR and slightly degrades LCP / TTI on cold loads.
- **Evidence:** raw counts from the route sweep (n=22 routes, all
  routes 200, all redirects clean):
  ```
  pageError counts:
    22 × Minified React error #418
  ```
  Probable culprit (not verified — see "Suspected cause"): the
  i18next + Sentry init in `app/entry.client.tsx` runs
  `await initI18nClient()` *before* `hydrateRoot()` is called. If
  the SSR-rendered HTML used resolved translation strings but the
  client's first hydration pass renders with the un-resolved fallback
  (or vice-versa, depending on whether the resolved namespace was
  inlined into the HTML), the mismatch shows up uniformly on every
  route. The empty `<title>Loading...</title><title>Login — Pickup
  Roster</title>` double-title in the `/api/healthz` redirect HTML
  body (see P0 above) suggests a similar render-once-then-rewrite
  pattern is in play.
- **Reproduction:**
  ```ts
  test.describe("no hydration mismatches", () => {
    for (const path of ["/", "/pricing", "/login", "/signup?plan=car-line"]) {
      test(`pageerror-free: ${path}`, async ({ page }) => {
        const errs: string[] = [];
        page.on("pageerror", (e) => errs.push(e.message));
        await page.goto(path, { waitUntil: "domcontentloaded" });
        await page.waitForLoadState("networkidle").catch(() => {});
        expect(errs, errs.join("\n")).toEqual([]);
      });
    }
  });
  ```
- **Suspected cause:** i18n hydration timing in
  `app/entry.client.tsx` (the `await initI18nClient()` before
  `hydrateRoot`) OR a server/client rendering divergence in `root.tsx`
  for locale/timezone-dependent strings. Inlining the resolved i18n
  namespace into the HTML and skipping the client-side `await` should
  resolve it; alternatively, defer the mismatched subtree behind a
  `<ClientOnly>` / `useHydrated` guard.
- **Status:** open — fires uniformly site-wide, so the fix is one
  shared cause, not 22 separate fixes.

### [P1] CSP blocks YouTube embeds on the homepage

- **Where:** `GET /` (and `/viewer-access` which redirects to `/`)
- **What:** The homepage attempts to embed
  `https://www.youtube-nocookie.com/...` in an `<iframe>`, but the
  staging CSP `frame-src` directive only allows
  `https://js.stripe.com https://checkout.stripe.com
  https://billing.stripe.com`. The browser blocks the frame and
  prints a CSP violation to the console for **each** embed
  (~6 occurrences per `/` load). User-visible: any video block on the
  marketing homepage renders blank.
- **Evidence:** console capture (deduped to one entry per page, real
  count is 6 per `/` load × 2 pages = 12 console errors total
  observed in the sweep):
  ```
  Framing 'https://www.youtube-nocookie.com/' violates the following
  Content Security Policy directive: "frame-src https://js.stripe.com
  https://checkout.stripe.com https://billing.stripe.com". The request
  has been blocked.
  ```
- **Reproduction:**
  ```ts
  test("homepage CSP allows youtube-nocookie embeds", async ({ page }) => {
    const csp: string[] = [];
    page.on("console", (m) => {
      if (m.type() === "error" && /Content Security Policy/i.test(m.text()))
        csp.push(m.text());
    });
    await page.goto("/");
    await page.waitForLoadState("networkidle").catch(() => {});
    expect(csp.filter((m) => /youtube-nocookie/.test(m))).toEqual([]);
  });
  ```
- **Suspected cause:** CSP `frame-src` allowlist in the worker
  middleware (likely `app/domain/utils/security-headers.server.ts` or
  similar) is missing `https://www.youtube-nocookie.com`. Either add
  it, or swap the embeds for a click-to-play poster image (lighter,
  also avoids the CSP+CMP overhead).
- **Status:** open — content was added to the marketing page without
  the matching CSP allowance.

### [P2] `__manifest` request is aborted twice during page transitions

- **Where:** `/` (initial load, after `requestfailed` on
  `/__manifest?paths=…&version=ea5d5abe`).
- **What:** Two
  `net::ERR_ABORTED https://…/__manifest?paths=%2F%2C%2Fblog%2C%2Ffaqs%2C%2Fguides%2C%2Flogin%2C%2Fpricing%2C%2Fsignup%2C%2Fstatus&version=ea5d5abe`
  requestfailed events on the homepage, paired with two
  `Failed to fetch manifest patches TypeError: Failed to fetch (at
  /assets/chunk-UVKPFVEO-KPIciH0D.js:12:7214)` console errors.
  Likely benign (RR fires speculative manifest prefetches that are
  cancelled when the user navigates), but it is noisy in the console
  and indistinguishable from a real network failure to a developer
  reading the panel.
- **Evidence:** see `/tmp/scan-findings.json` netFail entries; same
  URL appears twice on `/` and `/viewer-access` (which redirects to
  `/`), 0 occurrences on subsequent navigations within the sweep.
- **Reproduction:**
  ```ts
  test("homepage manifest prefetch is not aborted", async ({ page }) => {
    const aborted: string[] = [];
    page.on("requestfailed", (req) => {
      if (req.url().includes("/__manifest")) aborted.push(req.url());
    });
    await page.goto("/");
    await page.waitForLoadState("networkidle").catch(() => {});
    expect(aborted).toEqual([]);
  });
  ```
- **Suspected cause:** Speculative prefetch fired during the
  React-Router-7 idle-time manifest patch flow, then cancelled when
  the user lingers / interacts before it resolves. If confirmed
  benign, suppress with a console-error filter in
  `entry.client.tsx`'s Sentry integration; if not, debounce the
  prefetch trigger.
- **Status:** open, low priority.

- **Status:** open, low priority. Not a staging-deploy bug; an
