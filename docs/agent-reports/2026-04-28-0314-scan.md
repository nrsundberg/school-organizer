## Findings

### [P0] `staging.pickuproster.com` apex is being served as a tenant host (signup, home, healthz, onboarding all 302 → `/viewer-access`)

- **Where:** `https://staging.pickuproster.com/`,
  `/signup`, `/signup?plan=car-line`, `/district/signup`, `/api/healthz`,
  `/api/onboarding` — every route whose marketing-vs-tenant gating in
  `app/domain/utils/global-context.server.ts` depends on
  `onMarketingHost`.
- **What:** All of those routes return `302 Location: /viewer-access?next=…`
  on the canonical staging URL. The same routes return `200` on the
  workers.dev fallback (`school-organizer-staging.sundbergne.workers.dev`)
  in the same scan window, against the same Worker code. That mismatch
  means apex-domain traffic is being treated as a tenant subdomain
  (`isMarketingHost` returns false on apex), not as the marketing host.
- **Evidence:** raw HTTP probe — apex vs. workers.dev for the same paths:
  ```
  302  https://staging.pickuproster.com/                 -> /viewer-access?next=%2F
  302  https://staging.pickuproster.com/signup           -> /viewer-access?next=%2Fsignup
  302  https://staging.pickuproster.com/signup?plan=...  -> /viewer-access?next=%2Fsignup%3Fplan%3Dcar-line
  302  https://staging.pickuproster.com/district/signup  -> /viewer-access?next=%2Fdistrict%2Fsignup
  302  https://staging.pickuproster.com/api/healthz      -> /viewer-access?next=%2Fapi%2Fhealthz
  302  https://staging.pickuproster.com/api/onboarding   -> /viewer-access?next=%2Fapi%2Fonboarding

  200  https://school-organizer-staging.sundbergne.workers.dev/
  302  https://school-organizer-staging.sundbergne.workers.dev/api/healthz -> /login?next=%2Fapi%2Fhealthz
  400  https://school-organizer-staging.sundbergne.workers.dev/api/onboarding   (real validation 400, not auth gate)
  ```
  Browser repro from `flow.json` — Playwright lands on
  `viewer-access?next=%2Fsignup%3Fplan%3Dcar-line%26cycle%3Dmonthly`
  after clicking the Pricing → "car-line" plan link, with no email
  input rendered. `fail-signup-step1-loaded.png` captures the moment.
- **Reproduction:** `curl -I https://staging.pickuproster.com/`. Expected
  `200 text/html` (marketing home). Actual `302 Location: /viewer-access`.
- **Suspected cause:** routing collision between the production worker
  and the staging worker on the `pickuproster.com` zone. `wrangler.jsonc`
  pins prod under `env.production.routes` with a `*.pickuproster.com/*`
  wildcard:

  ```jsonc
  // wrangler.jsonc env.production.routes (lines 168–181)
  { "pattern": "pickuproster.com",     "custom_domain": true },
  { "pattern": "www.pickuproster.com", "custom_domain": true },
  { "pattern": "*.pickuproster.com/*", "zone_name": "pickuproster.com" }
  ```

  The wildcard matches `staging.pickuproster.com`, which in turn is
  also claimed by `env.staging.routes` as a Custom Domain. Cloudflare
  appears to be sending the apex traffic to the **production** worker
  (whose `PUBLIC_ROOT_DOMAIN=pickuproster.com` and whose
  `MARKETING_HOSTS` does NOT include `staging.pickuproster.com`).
  `isMarketingHost` therefore evaluates `staging.pickuproster.com` as
  `staging`-slug under root `pickuproster.com` — a tenant host — and
  the anonymous-skip-viewer disjunction in
  `app/domain/utils/global-context.server.ts:252-263` falls through to
  the `/viewer-access` redirect at line 277.

  Workers.dev URL works because it only routes to the staging worker;
  no prod-route ambiguity.

  Note the prior commit `818f54af fix(staging): env.staging.routes=[]
  so wrangler deploy stops fighting prod for pickuproster.com routes`
  was a partial mitigation of this exact class of collision; the
  follow-on `674e2527 feat(staging): mirror production on
  *.staging.pickuproster.com` re-introduces the staging routes but
  doesn't carve `staging.pickuproster.com` out of the prod wildcard.
- **Status:** open. Possible fixes (all on the prod side, not the
  staging side):
  - Tighten the prod wildcard so it does NOT match the `staging`
    label — e.g. add a Cloudflare Page Rule / route exclusion that
    short-circuits `staging.pickuproster.com` and
    `*.staging.pickuproster.com` away from the prod worker.
  - Move staging onto a non-overlapping zone (e.g.
    `pickuproster-staging.com`).
  - Or: add `staging.pickuproster.com` to the **production** worker's
    `MARKETING_HOSTS` env so even if the prod worker handles it, the
    marketing routes load. This is a workaround, not a real fix —
    prod would then own staging traffic and use the prod D1 / R2 /
    queue bindings.

### [P0] `GET /api/healthz` still does not return a JSON 200 (carry-over from 2026-04-23-2317, 2026-04-24-0001, 2026-04-26-0320, 2026-04-27-0318)

- **Where:** `GET /api/healthz` against the staging worker (any URL).
- **What:** Unchanged from prior scans — the healthcheck endpoint is
  gated by the same auth middleware that protects tenant routes:
  - On apex: `302 Location: /viewer-access?next=%2Fapi%2Fhealthz`
  - On workers.dev: `302 Location: /login?next=%2Fapi%2Fhealthz`

  Either way, an external uptime monitor / Cloudflare load-balancer
  origin-health check / k8s-style readiness probe will follow the
  redirect, hit a rendered HTML 200 from `/login` or `/viewer-access`,
  and report the worker as healthy without ever validating runtime
  state. The endpoint is effectively a no-op probe.
- **Evidence:**
  ```
  $ curl -sI https://staging.pickuproster.com/api/healthz
  HTTP/2 302
  location: /viewer-access?next=%2Fapi%2Fhealthz

  $ curl -sI https://school-organizer-staging.sundbergne.workers.dev/api/healthz
  HTTP/2 302
  location: /login?next=%2Fapi%2Fhealthz
  ```
- **Reproduction:** trivial — `curl -I` against either URL.
- **Suspected cause:** unchanged — `publicMarketingPath` allowlist in
  `app/domain/utils/global-context.server.ts` does not include
  `/api/healthz`. The fix for `/status` in 2026-04-25-0001 added
  exactly the same one-line allowlist entry; the same pattern applies.
- **Status:** still open. One-line fix lives on
  `agent-fix/2026-04-27-healthz-public-allowlist` (commit `dd688b0e`):
  adds `pathname === "/api/healthz"` to `publicMarketingPath`. Branch
  has been pushed but not merged — staging-deploy gate blocked the
  prior fixer because `CLOUDFLARE_API_TOKEN` was not present in this
  bash sandbox's env.

### [P1] CSP `frame-src` blocks the marketing-home YouTube embed

- **Where:** `https://school-organizer-staging.sundbergne.workers.dev/`
  (and `https://staging.pickuproster.com/` once the apex P0 is fixed).
- **What:** The marketing home renders an iframe pointing at
  `https://www.youtube-nocookie.com/embed/M7lc1UVf-VE` (3 instances
  found in the rendered HTML), but the response's CSP header is:

  ```
  Content-Security-Policy: …
    frame-src https://js.stripe.com https://checkout.stripe.com https://billing.stripe.com;
  ```

  Browser refuses to load the iframe and logs three console errors per
  page load:
  ```
  Refused to frame 'https://www.youtube-nocookie.com/' because it violates
  the following Content Security Policy directive: "frame-src
  https://js.stripe.com https://checkout.stripe.com https://billing.stripe.com".
  ```

  Net effect: the explainer video on the marketing home doesn't render
  for any visitor.
- **Evidence:** `page.on("console")` capture from `flow.json`:
  ```
  Framing 'https://www.youtube-nocookie.com/' violates the following
  Content Security Policy directive: "frame-src https://js.stripe.com
  https://checkout.stripe.com https://billing.stripe.com". The request
  has been blocked.
  ```
  Source HTML excerpt (from `fetch` against
  `https://school-organizer-staging.sundbergne.workers.dev/`):
  ```
  youtube-nocookie.com/embed/M7lc1UVf-VE
  youtube-nocookie.com/embed/M7lc1UVf-VE
  youtube-nocookie.com/embed/M7lc1UVf-VE
  ```
- **Reproduction:** open the home in a browser with devtools console
  open; observe three CSP violations. Or `curl` the page, find the
  iframe, then `curl -I` the same URL and grep the CSP `frame-src`.
- **Suspected cause:** the CSP middleware (probably in
  `app/lib/csp.server.ts` or similar) hard-codes the Stripe family for
  `frame-src` and was never extended when the YouTube embed landed on
  the marketing home. Should add
  `https://www.youtube-nocookie.com https://www.youtube.com` to
  `frame-src`.
- **Status:** open.

### [P1] React minified error #418 (hydration text mismatch) on every marketing page

- **Where:** every marketing page on both URLs — confirmed on
  `/pricing` (apex), `/` (workers.dev),
  `/viewer-access` (apex). Almost certainly on `/faqs`, `/blog`,
  `/guides` too — same root layout.
- **What:** Browser raises a `pageerror`:
  ```
  Error: Minified React error #418; visit
  https://react.dev/errors/418?args[]=text&args[]= for the full message
  or use the non-minified dev environment for full errors and
  additional helpful warnings.
  ```
  React error #418 is "Hydration failed because the server rendered
  HTML didn't match the client" — specifically the `text` variant,
  meaning a text node differs between SSR output and client render.
  React falls back to a full client re-render, which throws away SSR's
  performance win and can cause flicker / layout shift.
- **Evidence:** captured on three independent navigations
  (`flow.json:pageErrors[0]`, `scan-pricing.mjs` capture above,
  workers.dev parity run). Identical error string in all three —
  rules out a per-route bug.
- **Reproduction:** load any marketing page in a real browser with
  devtools open; the error logs once per navigation.
- **Suspected cause:** something in the marketing layout renders a
  client-only value during SSR. Common culprits in this codebase:
  - A `Date.now()` / `new Date()` formatted in `root.tsx` or a header
    component (clock, "as of" stamp).
  - The `i18next` language detector resolving to a different locale
    on server vs. client (this codebase has an active i18n branch and
    the prior 2026-04-27-0318 scan flagged a related `i18n flash`).
  - A `useId`-based collision after the i18n upgrade.
  - A Sentry/RUM script that mutates the body before hydration.

  Ranking these isn't possible from the minified error alone; a
  follow-up run with a non-minified build (set
  `NODE_ENV=development` in the staging deploy temporarily, or grab
  the matching sourcemap from the build output) is the cheapest way
  to localize.
- **Status:** open.

### [P2] Cloudflare RUM beacon (`/cdn-cgi/rum`) request fails with `net::ERR_ABORTED`

- **Where:** every page load on staging where the Cloudflare RUM
  script auto-injects.
- **What:** Browser logs a failed request:
  ```
  POST https://staging.pickuproster.com/cdn-cgi/rum?  ->  net::ERR_ABORTED
  ```
  The CSP `connect-src` directive is:
  ```
  connect-src 'self' https://*.sentry.io https://*.ingest.sentry.io https://api.stripe.com;
  ```
  `'self'` covers the apex host but the Cloudflare beacon endpoint
  uses an internal `cdn-cgi` path that is then aborted, possibly by
  CSP, possibly by the browser cancelling the unload-time fetch.
  No user-visible impact — the beacon is performance telemetry only.
  Reported here mainly because it pollutes any console-error monitor
  the team wires up later.
- **Evidence:** `flow.json:requestFailed[0]`.
- **Reproduction:** load any page; observe the failed POST to
  `/cdn-cgi/rum?` in the network tab.
- **Suspected cause:** Cloudflare's RUM is enabled at the zone level
  but the beacon either races the page-unload (typical
  ERR_ABORTED cause) or is blocked by some other middleware. Could
  also be safe to disable in the staging zone since prod-traffic
  metrics are what matter.
- **Status:** open. Lower-bound P3 if the team decides to disable
  RUM on staging entirely.
