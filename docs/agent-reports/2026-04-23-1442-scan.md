The UI then renders the inline string "Unable to create account." under the Password row and stays on step 1 ‚Äî no advance to step 2, no email sent, no tenant created.

  ```

    -X POST https://school-organizer-staging.sundbergne.workers.dev/api/auth/sign-up/email \
  ```

- **Reproduction (UI flow):** visit `/` ‚Üí click **Pricing** ‚Üí click **Start Free Trial** under Car Line ‚Üí fill step 1 ‚Üí click **Continue** ‚Üí UI shows "Unable to create account." and URL stays at `/signup?plan=car-line`. Console logs `Failed to load resource: the server responded with a status of 500 ()` for the `/api/auth/sign-up/email` request. No page-level (uncaught) errors.
- **Evidence captured:**
  - Flow test stdout: `FLOW_BAD_REQS=[...{"url":".../api/auth/sign-up/email","status":500,"failure":null}]`
  - Final URL after Continue: `‚Ä¶/signup?plan=car-line` (did not advance)
  - Body text contained `Unable to create account.`
  - Screenshots: `/sessions/<scan-sandbox>/flow-step1.png` and `flow-step2.png` (in the scanner sandbox's scratch dir, not committed ‚Äî signup step 1 pre-submit and post-submit)
- **Severity:** P0. This is a brand-new user's first impression ‚Äî the "Start Free Trial" button on every pricing card ends in a 500. Nothing downstream (step 2 ‚Üí Stripe checkout ‚Üí tenant creation) can be exercised until this is fixed.
- **Likely area to investigate:** `better-auth` email/password signup handler (or whatever wraps `/api/auth/sign-up/email` on the Worker), most likely a binding/env that exists in prod/dev but is missing or misconfigured in `wrangler.jsonc > env.staging` (shared secret, D1 binding, email queue binding, etc.). ### [P1] `/blog` on staging redirects to `/login` despite being advertised as a public marketing link
- **Where:** `GET https://school-organizer-staging.sundbergne.workers.dev/blog`
- **What:** Every surface on the marketing site (the top nav on `/`, `/pricing`, `/signup?plan=‚Ä¶`, `/faqs`; the footer on every one of those; and the `content/blog/` section) exposes a `<a href="/blog">Blog</a>` link. Hitting that URL on staging returns `HTTP/2 302 Location: /login?next=%2Fblog`. `/blog/` (with trailing slash) redirects the same way (`/login?next=%2Fblog%2F`). A signed-out marketing visitor following the Blog link is punted to a login form, not shown any content.
- **Reproduction:**
  ```
  curl -sI https://school-organizer-staging.sundbergne.workers.dev/blog | head -3
  # HTTP/2 302
  # location: /login?next=%2Fblog
  ```
- **Severity:** P1. It's not a 5xx and it doesn't break a flow you already paid for, but it's broken marketing on the public site ‚Äî the most recent commit `c8f0685 feat(blog): marketing blog index + post routes` added blog routes, and `/blog` is present in both the primary nav and the footer on every rendered marketing page. This is visible content rot, not a minor copy issue.
- **Likely area to investigate:** host-based routing in `app/` (React Router v7 framework mode). Everything else public on staging ‚Äî `/`, `/pricing`, `/faqs` ‚Äî is correctly reachable on the `‚Ä¶workers.dev` "marketing host". `/blog` must be mounted behind a loader that requires a session, or the route is guarded by a middleware that doesn't whitelist `/blog` on the marketing host. Compare with `/faqs`, which is clearly allowed.
- **Not covered by smoke:** `e2e/smoke-routes.ts > publicMarketingRoutes` lists `/`, `/pricing`, `/faqs`, `/status`, `/login`, `/signup?plan=car-line`, `/forgot-password`, `/reset-password` ‚Äî no `/blog` entry. Adding `/blog` to the list would have caught this on the previous smoke run.

### [P2] Signup form labels are not associated with their inputs (step 1 of `/signup`)
- **Where:** `GET /signup?plan=car-line` ‚Äî the "Create your account" form (`<label>Your name</label>`, `Email`, `Phone number`, `Password`, `Confirm password`).
- **What:** Each field is rendered as `<label class="‚Ä¶">Your name</label><input ‚Ä¶/>` with no `for`/`id` association and the input is not nested inside the label. Screen-reader users will hear "text edit, blank" (or whatever the placeholder says, depending on AT) for every field, and Playwright's `page.getByLabel("Your name")` / `getByLabel("Email")` can't find them either ‚Äî which is why `e2e/auth.spec.ts` `/signup renders an org name / school name field` has been failing independently of the P0 above.
- **Evidence:** from the staging HTML of `/signup?plan=car-line`:
  ```html
  <label class="text-sm text-white/80">Your name</label>
  <input class="input input--primary" data-slot="input" type="text"
         placeholder="Jane Coach" required autoComplete="name" data-rac="" value=""/>
  <label class="text-sm text-white/80">Email</label>
  <input class="input input--primary" data-slot="input" type="email"
         placeholder="you@school.edu" required autoComplete="email" data-rac="" value=""/>
  ```
  None of the `<input>`s carry an `id`, none of the `<label>`s carry a `for`, and the labels are siblings, not ancestors.
- **Severity:** P2. This is a real WCAG 2.1 A failure (1.3.1 Info and Relationships, 4.1.2 Name/Role/Value) on one of the highest-traffic forms on the site ‚Äî customer-facing, prospecting ‚Äî and it also makes the form fragile under the existing auth Playwright assertions.
- **Affects tests:** `e2e/auth.spec.ts` `/login renders an email field and submit button` and `/signup renders an org name / school name field` both use `getByLabel`. On `/login` the smoke still passes (login has single-field focus + submit), but the `/signup` test fails here; see "Stale tests" below for the separate redirect issue on that same spec.

### [P3] Homepage wordmark link accessible name is "PickupRoster" (no space), not "Pickup Roster"
- **Where:** top-level nav on `/` and other marketing pages. The element:
  ```html
  <a aria-label="PickupRoster home" href="/">
    <img src="/logo-wordmark.svg" alt="PickupRoster" class="h-8 w-auto"/>
  </a>
  ```
- **What:** The product name everywhere in copy is "Pickup Roster" (two words) ‚Äî page `<title>`, footer paragraph, hero subhead, pricing blurbs. Only the wordmark's `aria-label` and the image `alt` collapse it to "PickupRoster". This is also why `e2e/marketing.spec.ts` `homepage loads with Pickup Roster nav link` fails: `getByRole('link', { name: 'Pickup Roster' })` can't match the `aria-label="PickupRoster home"`.
- **Severity:** P3. Cosmetic/a11y consistency, not user-blocking. Fix is one word ‚Äî align the `aria-label` and `alt` to "Pickup Roster".
