## Findings

### [P0] `GET /api/healthz` still redirects to `/login` (carry-over from 2026-04-23-2317, 2026-04-24-0001, 2026-04-26-0320)

- **Where:** `GET /api/healthz` against staging.
- **What:** The healthcheck endpoint is gated by the same auth
  middleware that protects tenant routes — an unauthenticated request
  receives `302 Location: /login?next=%2Fapi%2Fhealthz` instead of a
  JSON 200. Any external uptime monitor / Cloudflare load-balancer
  origin-health check / k8s-style readiness probe will follow the
  redirect, hit `/login`, and report the worker as healthy on a
  rendered HTML 200, never actually validating the worker's runtime
  state. The endpoint is effectively a no-op.
- **Evidence:** raw HTTP probe (no browser, no cookies, no session):
  ```
  $ curl -sI https://school-organizer-staging.sundbergne.workers.dev/api/healthz
  HTTP/2 302
  location: /login?next=%2Fapi%2Fhealthz
  ```
  Reproduced from this run's `http-probe.txt`:
  ```
  302 	 51ms 	 /api/healthz 	 /login?next=%2Fapi%2Fhealthz
  ```
- **Reproduction:** trivial — `curl -I` against the staging URL.
- **Suspected cause:** unchanged from prior scans —
  `publicMarketingPath` allowlist in
  `app/domain/utils/global-context.server.ts` (or wherever the auth
  redirect middleware lives) does not include `/api/healthz` (or
  `/api/*`). The `/status` fix in 2026-04-25-0001 added the same
  one-line allowlist entry for `/status`; the same pattern applies
  here.
- **Status:** open — fourth consecutive scan. The fixer's 2026-04-25
  pass updated `/status` but did not generalize to `/api/healthz`.

### [P1] Raw i18n translation keys are visible on first paint of every public route

- **Where:** `/`, `/pricing`, `/signup?plan=car-line` (almost certainly
  every route — these three were sampled). Signup is the most severe
  example because a Continue button literally renders as
  `signup.step1.continue`.
- **What:** The SSR'd HTML embeds raw i18next keys
  (e.g. `marketingNav.pricing`, `signup.step1.title`,
  `signup.stepper.account`, `footer.nav.pricing`,
  `signup.step1.continue`) directly inside the markup. On a cold cache
  the user sees these untranslated keys for the time it takes the
  client-side i18n bundle to fetch and apply (`/locales/en/*.json`),
  typically a few hundred ms but unbounded on a slow connection. SEO
  impact is real too: anything that reads the HTML before JS executes
  (Googlebot mobile, Bing, social-media unfurlers) sees the keys.
  This is also the most likely root cause of the React #418 hydration
  mismatch reported separately below — server renders the bare key,
  client hydrates the translated string, React aborts.
- **Evidence:**
  1. Fresh `chromium` launch + `goto(...,{waitUntil:"domcontentloaded"})`
     captures of `body.innerText` (i18n-probe.json):
     ```
     /                       earlyKeys: marketingNav.pricing,
                                       marketingNav.blog,
                                       marketingNav.guides,
                                       marketingNav.faqs,
                                       marketingNav.login,
                                       marketingNav.signup,
                                       footer.nav.pricing,
                                       footer.nav.blog,
                                       footer.nav.faqs,
                                       footer.nav.login
                             lateKeys (after networkidle): []
     /pricing                earlyKeys: ...marketingNav...,
                                       pricing.heading,
                                       pricing.lede,
                                       pricing.cycle.monthly,
                                       pricing.cycle.annual
                             lateKeys: []
     /signup?plan=car-line   earlyKeys: ...marketingNav...,
                                       signup.stepper.account,
                                       signup.stepper.school,
                                       signup.stepper.plan,
                                       signup.step1.title
                             lateKeys: []
     ```
  2. Direct `curl` of the SSR HTML for `/signup?plan=car-line` finds
     these keys in the raw response body (no JS executed):
     ```
     $ curl -s https://school-organizer-staging.sundbergne.workers.dev/signup?plan=car-line \
         | grep -oE '(signup|marketingNav|footer\.nav)\.[a-zA-Z.]+' \
         | sort -u | head
     footer.nav.blog
     footer.nav.faqs
     footer.nav.login
     footer.nav.pricing
     footer.nav.status
     footer.nav.support
     signup.step
     signup.stepper.account
     signup.stepper.plan
     signup.stepper.school
     ```
  3. Screenshot `v2-step1-filled.png` (in scan probe dir) shows the
     entire signup-step-1 form rendering with raw keys for every
     label *and* the Continue button — not theoretical, this happens
     in the first ~hundreds of ms of every fresh visit.
- **Reproduction:**
  ```ts
  test("first paint renders translated strings, not i18n keys", async ({ page }) => {
    await page.goto("/signup?plan=car-line", { waitUntil: "domcontentloaded" });
    const text = await page.evaluate(() => document.body.innerText);
    expect(text, "no raw i18n keys visible at DOMContentLoaded")
      .not.toMatch(/\b(signup|marketingNav|footer\.nav)\.[a-z]/i);
  });
  ```
- **Suspected cause:** SSR-side i18next is not initialized with the
  loaded resource bundle before the React tree is rendered to string.
  The keys serialize as their literal names because `t("signup.step1.title")`
  returns its key when no translation is registered. Two plausible
  fixes: (a) bundle the `en` namespace into the worker so SSR has it
  synchronously, or (b) add a `useTranslation`-aware render-blocker
  that defers SSR output until namespaces are ready. Option (a) is
  the cheaper one given the JSON files are already in the repo.
- **Status:** open — new finding (prior scans only checked text after
  `networkidle`, which masks this entirely).

### [P1] React error #418 (hydration mismatch) on `/signup?plan=car-line`

- **Where:** `/signup?plan=car-line` (prior scans observed it on
  every public route; this run only navigated to `/`, `/pricing`,
  `/signup` and saw it fire on `/signup`).
- **What:** Each fresh load of `/signup?plan=car-line` raises
  ```
  Error: Minified React error #418; visit
  https://react.dev/errors/418?args[]=text&args[]= ...
  ```
  in the browser's `window.onerror` channel. React #418 is the
  "Hydration failed because the server rendered text didn't match
  the client" error. Effect: the page's interactive root is fully
  re-rendered from scratch on the client (loses any in-flight focus,
  burns one extra render budget on every navigation, and any
  hand-written `useEffect`s that assumed hydration completed are
  triggered twice).
- **Evidence:** captured in two separate runs of this scan:
  - `flow-result.json` ➜ `page_errors`: 2 occurrences of #418 during
    `/` → `/pricing` → `/signup` traversal.
  - `v2-result.json` ➜ `page_errors`: 1 occurrence on the standalone
    fresh load of `/signup?plan=car-line`.
- **Reproduction:**
  ```ts
  test("/signup?plan=car-line hydrates without #418", async ({ page }) => {
    const errs: string[] = [];
    page.on("pageerror", (e) => errs.push(e.message));
    await page.goto("/signup?plan=car-line");
    await page.waitForLoadState("networkidle").catch(() => {});
    expect(errs.filter((m) => /Minified React error #418/.test(m))).toEqual([]);
  });
  ```
- **Suspected cause:** Almost certainly the same root cause as the
  i18n-key flash above — SSR renders the raw key, client renders the
  translated string, hydration aborts. Fixing the i18n SSR
  initialization should make this finding resolve as a side effect.
  If it doesn't, the next candidate is the locale negotiation
  divergence between `entry.server.tsx` and `entry.client.tsx`.
- **Status:** open — carry-over from 2026-04-26-0320 (P1). Symptom
  unchanged.

### [P1] CSP `frame-src` still blocks the homepage YouTube embed (carry-over from 2026-04-26-0320)

- **Where:** `GET /` (homepage demo video block).
- **What:** The homepage ships an `<iframe src="https://www.youtube-nocookie.com/...">`
  element, but the staging CSP `frame-src` directive only allows
  `https://js.stripe.com https://checkout.stripe.com https://billing.stripe.com`.
  Chromium blocks the frame and prints
  ```
  Framing 'https://www.youtube-nocookie.com/' violates the following
  Content Security Policy directive: "frame-src https://js.stripe.com
  https://checkout.stripe.com https://billing.stripe.com". The request
  has been blocked.
  ```
  to the console once per embed (3 embeds × 2 page loads in this run
  = 6 console errors). The user-visible result is a blank rectangle
  where the demo video should play.
- **Evidence:**
  1. `flow-result.json` console_errors[0..5] all match the message
     above with `youtube-nocookie.com` framed.
  2. Direct grep of the SSR HTML for `/` confirms 3 separate
     `youtube-nocookie.com` references:
     ```
     $ curl -s https://...workers.dev/ | grep -oE 'youtube-nocookie\.com[^"]+'
     youtube-nocookie.com/ ... (3 hits)
     ```
- **Reproduction:** identical to the prior scan's reproduction (same
  finding, no change):
  ```ts
  test("homepage CSP allows youtube-nocookie embeds", async ({ page }) => {
    const csp: string[] = [];
    page.on("console", (m) => {
      if (m.type() === "error" && /Content Security Policy/i.test(m.text()))
        csp.push(m.text());
    });
    await page.goto("/");
    await page.waitForLoadState("networkidle").catch(() => {});
    expect(csp.filter((m) => /youtube-nocookie/.test(m))).toEqual([]);
  });
  ```
- **Suspected cause:** unchanged — `frame-src` allowlist in the worker
  CSP middleware (likely `app/domain/utils/security-headers.server.ts`)
  is missing `https://www.youtube-nocookie.com`. Either add it, or
  swap the embed for a click-to-play poster.
- **Status:** open — unchanged from 2026-04-26-0320 P1. No fix
  shipped between the two runs.
