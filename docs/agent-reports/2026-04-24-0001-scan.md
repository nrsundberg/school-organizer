## Findings

### [P0] `/api/healthz` still redirects (carry-over from 2026-04-23-2317)

- **Where:** `GET /api/healthz`
- **What:** `web_fetch` against `/api/healthz` returns `Fetch failed:
  Redirect was cancelled` — same behavior the prior P0 documented
  (302 → `/login?next=%2Fapi%2Fhealthz`). External uptime monitors and
  the smoke suite's `e2e/smoke.spec.ts:105` both rely on a JSON
  `{ ok: true }` response at this path; the route is still gated by the
  unauthenticated-redirect middleware.
- **Evidence:** HTTP probe response (full):
  ```
  GET https://school-organizer-staging.sundbergne.workers.dev/api/healthz
  → Fetch failed: Redirect was cancelled
  ```
  No 2xx/JSON body returned. Identical failure mode to the
  2026-04-23-2317-scan reproduction.
- **Reproduction:**
  ```ts
  const r = await request.get("/api/healthz", {
    maxRedirects: 0,
    failOnStatusCode: false,
  });
  expect(r.status()).toBe(200);
  expect(await r.json()).toMatchObject({ ok: true });
  ```
- **Suspected cause:** Same as before — `publicMarketingPath` in
  `app/domain/utils/global-context.server.ts` does not list `/api/*`.
  No fix appears to have shipped to staging since 2026-04-23-2317
  (deployed manifest version unchanged; 2026-04-24 nightly build is
  BLOCKED on `wrangler` auth per `docs/nightly/2026-04-24-build.md`).
- **Status:** open — believed unchanged from prior scan. Please confirm
  with a Playwright run after the staging-deploy gate is unblocked.

### [P2] Sentry tracing-integration warning — unverifiable this run, likely not firing on staging

- **Where:** every public marketing route on staging (per the prior
  scan — `/`, `/pricing`, `/faqs`, `/signup?plan=car-line`, `/login`,
  `/forgot-password`).
- **What:** Prior scan reported a console warning
  `browserTracingIntegration is not fully compatible with @sentry/react-router. Please use reactRouterTracingIntegration instead.`
- **Evidence (this run):** Console output is not reachable from
  HTTP-only probes. **Notable:** the inline streamed loader data on
  every probed page now serializes `"sentryDsn":""` (empty string) — see
  e.g. the `/`, `/pricing`, `/login`, `/forgot-password`, `/blog`,
  `/faqs` HTML bodies. With an empty DSN the Sentry browser SDK should
  not initialize on staging, which would suppress this warning here
  even though the underlying integration import is still wrong. The
  warning will reappear in any environment with a real DSN (prod, or
  staging once Sentry is wired) until the integration is swapped to
  `reactRouterTracingIntegration` in the Sentry init (likely
  `app/entry.client.tsx`).
- **Reproduction (when DSN is set):** Open any marketing route, check
  devtools console.
- **Status:** open — same fix recommended as in the prior scan; just
  notably won't be observable on this staging URL while
  `branding.sentryDsn` is empty.
