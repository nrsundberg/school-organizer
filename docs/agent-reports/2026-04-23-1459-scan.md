### [P2] Duplicate `<title>` tag on every page; the first one is "Loading..."
- **Where:** every route rendered by the React Router app — confirmed on
  `/`, `/pricing`, `/login`, `/signup?plan=car-line`.
- **What:** The document `<head>` emits two adjacent `<title>` elements,
  e.g. on `/`:
  ```
  <title>Loading...</title><title>Pickup Roster — Car line made clear</title>
  ```
  HTML5 allows only one `<title>` per document. Browsers render the first
  one they encounter, so the accessible name for the tab is "Loading..."
  on the initial SSR paint; screen readers announce the tab as
  "Loading...", search engines may treat "Loading..." as the page title
  for any URL they fetch, and Chrome tab history will show "Loading..."
  for every visit. This is a quality-of-experience and SEO regression
  across the whole marketing site.
- **Evidence:**
  ```
  curl -s https://school-organizer-staging.sundbergne.workers.dev/ \
    | grep -oE '<title[^>]*>[^<]*</title>'
  <title>Loading...</title>
  <title>Pickup Roster — Car line made clear</title>
  ```
  Same pattern on `/login` (`<title>Login — Pickup Roster</title>`) and
  `/pricing` (`<title>Pricing — Pickup Roster</title>`).
- **Reproduction:** curl any route, grep for `<title`.
- **Suspected cause:** A root loader or a `<title>` in `app/root.tsx` is
  emitting "Loading..." and the per-route `meta()` is appending a second
  one instead of replacing it. This used to be a common React Router v7
  pitfall when a route exports `meta` *and* the root renders a literal
  `<title>` in its JSX.
- **Status:** open.

### [P2] Signup form inputs lack programmatic labeling (accessibility)
- **Where:** `/signup?plan=car-line` (step 1).
- **What:** The five form inputs on the signup step-1 form (Your name,
  Email, Phone number, Password, Confirm password) are rendered as
  unlabeled `<input>` elements — no `id`, no `name`, no `aria-label`, and
  the adjacent `<label class="text-sm text-white/80">` elements have no
  `for`/`htmlFor` attribute. A screen reader focusing on any of these
  fields announces it only as "edit text, required" with no field name.
  Playwright's `getByLabel(/^Your name/i)` times out on this exact form
  because the label isn't programmatically associated.
- **Evidence:**
  ```
  $ curl -s .../signup?plan=car-line | grep -oE '<label[^>]*>[^<]*</label>' | head
  <label class="text-sm text-white/80">Your name</label>
  <label class="text-sm text-white/80">Email</label>
  <label class="text-sm text-white/80">Phone number</label>
  <label class="text-sm text-white/80">Password</label>
  <label class="text-sm text-white/80">Confirm password</label>

  $ curl -s .../signup?plan=car-line | grep -oE '<input[^>]*>' | head
  <input class="input input--primary" ... type="text"    placeholder="Jane Coach"  ... />
  <input class="input input--primary" ... type="email"   placeholder="you@..."     ... />
  <input class="input input--primary" ... type="tel"     placeholder="(555)..."    ... />
  <input class="input input--primary" ... type="password" placeholder="At least 8" ... />
  <input class="input input--primary" ... type="password" placeholder="Repeat"     ... />
  ```
  Labels have no `for=`, inputs have no `id=` / `name=`.
- **Reproduction:** Load the form, tab through the fields with VoiceOver
  or NVDA — each field announces as unlabeled.
- **Suspected cause:** The signup-step-1 form was built with a custom
  `<Input>` component (likely the React Aria primitive visible from
  `data-rac=""`), and the component was used with a plain `<label>`
  sibling rather than wrapping the input inside the label or passing
  `htmlFor` / `id`. Same regression likely applies to login, forgot
  password, and any other form sharing the primitive.
- **Status:** open.

### [P3] `<iframe allow="web-share">` on YouTube embeds is rejected by Chromium
- **Where:** Homepage hero — three `<iframe title="Getting started" | "Car
  line pickups" | "Fire drills and safety">` all emit
  `allow="... web-share"`.
- **What:** Chromium logs `Unrecognized feature: 'web-share'.` for each
  iframe. Harmless (Chromium ignores the unknown feature and the rest of
  the `allow` list still applies), but noisy in DevTools and a hint that
  the embed template pasted in was copied from a slightly newer YouTube
  embed spec than Chromium's current feature allowlist accepts. Quick to
  strip `web-share` from the list if we don't need it.
- **Evidence:** Browser console on `/`:
  ```
  [warning] Unrecognized feature: 'web-share'.
  ```
- **Reproduction:** Load `/` with DevTools console open.
- **Suspected cause:** Hard-coded `allow="..."` string on the YouTube
  `<iframe>` in `app/routes/_index.tsx` (or wherever the video embed
  component lives). Remove `web-share` from that string.
- **Status:** open.
